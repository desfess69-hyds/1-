## 0. 왜 3-Layer 아키텍처인가? (핵심 통찰)

**LLM은 확률적, 비즈니스는 결정적.**
모든 작업을 Claude에게 시키면 한 단계당 90% 성공해도 5단계 거치면 **59%**로 떨어진다.

**해결책**: 역할 분리
| 레이어 | 누가 | 무엇을 | 결과물 |
|--------|------|--------|--------|
| **1. Directive (지시서)** | 사람(서동현) | "이 일을 어떻게 할지" 한국어 설명 | `directives/*.md` |
| **2. Orchestration (판단)** | Claude (Code/API) | 지시서 읽고 → 도구 선택 → 순서 결정 → 에러 처리 | (실시간) |
| **3. Execution (실행)** | Python 스크립트 | API 호출, 이미지 생성, 업로드 등 반복 작업 | `execution/*.py` |

**왜 1인 회사에 완벽한가:**
- 코딩 초보 → directive는 한국어 마크다운으로 쓰면 됨
- Claude Code가 곧 오케스트레이터 → 별도 웹 UI 불필요
- 스크립트는 한번 만들면 영구 자산 → 시간 갈수록 시스템이 더 똑똑해짐
- 에러 발생 시 자가 강화(self-anneal) — 에러 → 스크립트 수정 → directive 업데이트 → 다음엔 안 틀림

## 1. HYDS가 만들 AI 에이전트 2개

### 1.1 수련회 기획·점검 AI
**역할**: 교회/단체가 의뢰하면 컨셉부터 체크리스트까지 AI가 초안을 다 만들고, 서동현이 검토·확정.

**서브 기능**:
- 기획안 생성 (컨셉·주제·말씀·프로그램·일정·예산·강사 후보·장소)
- 점검 체크리스트 자동 점검 (D-30, D-7, D-1, 당일, 사후)
- 위험 요소 사전 감지 (예산 초과, 강사 일정 충돌, 인원 미달 등)

### 1.2 인스타 카드뉴스 제작·업로드 AI
**역할**: 주제 한 줄만 주면 → 카피·이미지·해시태그·일정까지 → 인스타에 자동 업로드.

**서브 기능**:
- 카드뉴스 카피 작성 (1장당 1~2문장, 슬라이드 5~10장)
- 카드 이미지 생성 (Pillow로 텍스트+배경 합성, 통일된 브랜드 톤)
- 해시태그·캡션 자동 작성
- Instagram Graph API로 업로드 (수동 모드도 지원)

## 3. 첫 directive 예시 (한국어로 이렇게 쓰면 됨)

**`directives/plan_retreat.md`** — 수련회 기획 지시서

```markdown
# 수련회 기획 SOP

## 목표
교회/단체로부터 수련회 요청을 받으면 기획안 초안을 작성한다.

## 입력 (사용자가 줄 정보)
- 교회/단체명
- 대상 (예: 청년부 30명)
- 날짜 / 기간 (예: 2026-08-15 ~ 17, 2박 3일)
- 예산 (예: 300만원)
- 주요 메시지/방향 (선택)

## 사용할 도구
1. `execution/generate_retreat_plan.py` — Claude로 기획안 생성
2. `data/speakers.json` — 강사 풀 참조
3. `data/venues.json` — 장소 풀 참조

## 출력
`.tmp/dossiers/{교회명}_{날짜}_기획안.md` 파일로 저장.
양식: `templates/retreat_template.md` 따라가기.
필수 항목:
- 컨셉 한 줄
- 주제 말씀 (성경 본문 3안)
- 시간표 (일자별)
- 강사 후보 3명 (`speakers.json` 참조)
- 장소 후보 3곳 (`venues.json` 참조)
- 예산표 (강사료/장소/식대/교통/예비)

## 엣지 케이스
- 예산이 너무 적으면 → 솔직히 말하고 절충안 제안
- 인원 50명 이상이면 → 안전 관리(소방, 보험) 항목 추가
- 강사 풀이 비어있으면 → 사용자에게 강사 정보 요청

## 다음 단계
초안 완성 후 → 사용자가 검토 → `directives/review_retreat.md`로 점검 단계 진입
```

이런 식으로 **한국어 마크다운**으로 쓰면 끝. Claude Code가 이걸 읽고 알아서 실행.

| 영역 | 선택 | 이유 |
|------|------|------|
| 언어 | **Python 3.11+** | 스크립트 위주, Pillow·요청 라이브러리 풍부 |

| 저장소 | **JSON 파일 + .tmp/** | DB 미필요 — 1인 회사엔 충분 |
| Git | **GitHub (private)** | `.env`, `credentials.json`은 무조건 gitignore |

**Next.js는 안 씁니다.** 1인이 시스템 안착시키는 단계에선 웹 UI가 오히려 짐.

## 6. 자가 강화(Self-Anneal) 워크플로

```
작업 의뢰 → directive 따라 실행 → 에러 발생
                                      ↓
                              스크립트 수정 + 테스트
                                      ↓
                              directive에 "이런 경우엔 이렇게" 추가
                                      ↓
                              시스템이 1단계 더 똑똑해짐
```

**예시**: 카드뉴스 업로드 시 인스타 API가 "이미지 비율 안 맞음" 에러
→ `make_card_image.py`에 1:1 비율 강제 로직 추가
→ `directives/create_card_news.md`에 "이미지는 1080×1080 정사각형 필수" 명시
→ 다음부터 영원히 안 틀림

## 8. 핵심 원칙 (HYDS의 운영 철학)

1. **directive에 한국어로 쓴다** — 코딩 아니라 SOP. 누구나 쓸 수 있는 사용설명서.
2. **에러는 학습 기회** — 망하면 → 고치고 → directive에 적어둠 → 같은 에러는 두 번 안 만남.
3. **AI는 판단만, 반복은 스크립트** — 클로드한테 카드뉴스 이미지 만들어달라고 매번 부탁 X. Pillow 스크립트로 한 번 짜둠.
4. **로컬은 임시, 결과는 클라우드** — 최종 카드뉴스는 인스타로, 기획안은 노션/드라이브로.
5. **수익보다 시스템 먼저** — 한 번 만들어두면 평생 일함.
